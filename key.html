GitHub API 金鑰設定

1
正確位置

正確頁面：請前往 GitHub 個人設定中的 Personal Access Tokens 頁面：

https://github.com/settings/tokens

  1. 登入 GitHub 帳戶

    前往 GitHub 並使用您的帳號登入。

  2. 進入 Token 設定頁面

    點選右上角個人頭像，選擇 Settings，在左側選單中點選 Developer settings，再選擇 Personal access tokens。

提示：建議使用 GitHub Fine-grained tokens 以獲得更精細的權限管理，但也可依需求選擇傳統的 Token。

2
建立 GitHub Token 的步驟

  1. 建立新 Token

    點選 Generate new token（或 Fine-grained token 的選項）。

  2. 填寫 Token 資訊

    • Token 名稱：例如 auto-deploy-token 或 github-api-token（便於識別用途）。
    • 有效期限：建議設定為 90 至 180 天，以便定期更新提高安全性。
    • 選擇 Scopes（權限）：依照您的應用需求勾選合適的權限。

  3. 建立並複製 Token

    點選「Generate token」後，系統會生成一個 Token。

    務必複製並保存該 Token，因為它只會顯示一次。

3
GitHub 常見權限介紹

推薦勾選的 Scopes：

repo 主要

提供對所有儲存庫（私人與公共）的完整讀寫權限。適用於基本操作包括讀取、修改、推送、commit、branches、releases、merges 等。

workflow CI/CD

允許管理 GitHub Actions 的工作流程，包括讀取、修改與觸發流程。若需要自動部署或操作 CI/CD，必須勾選此項。

Pull requests 協作

管理 PR 的建立、審查與合併。讓 AI 能自動開啟、修改及合併 PR，完成程式碼合併流程。

Issues 協作

讀取與管理 Issues，包括留言、分派、標籤。便於 AI 追蹤專案問題，進行 Issue 的建立與管理。

Actions CI/CD

管理 workflow runs 與 artifacts。讓 AI 能查看與操作 CI/CD 相關結果和產生的 artifacts。

推薦設定：對於大部分自動部署與 AI 協作情境，建議至少開啟 repo 與 workflow 權限。

GitLab Personal Access Token 設定

1
正確建立 PAT 的位置

正確頁面： 請前往 GitLab 個人設定中的 Personal Access Tokens 頁面，正確的連結為：

https://gitlab.com/-/user_settings/personal_access_tokens

注意：不要誤點進專案內的「Access Tokens」或「Deploy Tokens」頁面，這些只針對單一專案或用於部署下載用途，無法跨專案存取。

2
建立 PAT 的步驟

  1. 登入 GitLab

    進入 https://gitlab.com 並登入您的帳號。

  2. 前往 PAT 設定頁面

    使用上述連結進入 Personal Access Tokens 頁面。

  3. 填寫 Token 資訊

    • Token 名稱：例如 claude-deploy-token 或 auto-deploy-token（便於識別用途）。
    • 過期日期：建議設定為 90 至 180 天，以便定期更新提高安全性。
    • 選擇 Scopes（權限）：建議只開啟必要的權限，遵循「最小權限原則」。

  4. 建立並複製 Token

    點選「Create personal access token」後，系統會生成一個 Token。

    務必複製並保存該 Token，因為它只會顯示一次。

3
GitLab 常見權限介紹

推薦勾選的 Scopes：

api 主要

提供完整的 API 存取權，包括所有群組、專案、容器註冊表、依賴代理與套件註冊表。

read_repository 讀取

允許讀取私人專案中的儲存庫內容。

write_repository 寫入

允許進行推送、提交與修改操作。

4
如何在專案中使用 PAT

將生成的 PAT 存入您的環境變數檔案（例如 .env），並確保該檔案不會公開上傳。示例格式如下：

GITLAB_PERSONAL_ACCESS_TOKEN=glpat-xxxxxxxxxxxxxxxxxxxx

並在版本控制中將 .env 加入 .gitignore，確保安全。

提示：可以在 .gitignore 中加入一行 .env 來排除該檔案。

# 使用 curl 測試 PAT curl --header "PRIVATE-TOKEN: glpat-xxxxxxxxxxxxxxxxxxxx" https://gitlab.com/api/v4/projects

Brave Search API 金鑰設定

1
正確位置

正確頁面：請前往 Brave API Dashboard 頁面：

https://api-dashboard.search.brave.com/app/keys

2
取得 Brave API 金鑰的步驟

  1. 登入與驗證

    輸入您的信箱地址，系統將發送驗證碼至您的電子郵件，請輸入驗證碼完成登入。

  2. 取得 API 金鑰

    登入後，您會在 Dashboard 中看到您的 API 金鑰。

    務必複製並保存該 API 金鑰，因為可能只顯示一次。

  3. 應用於專案設定

    將金鑰存入環境變數檔案中，例如：

    BRAVE_API_KEY=brv_XXXXXXXXXXXXXXXXXXXX
3
Brave Search API 注意事項
  • 速率限制

    注意 API 的使用限制，根據文件調整請求策略，避免超出限制。

  • 安全性

    務必將金鑰存放在安全位置，避免公開洩露。建議放入環境變數，不要硬編碼在應用程式中。

  • 簡單測試代碼
    # 使用 curl 測試 Brave Search API curl -X GET "https://api.search.brave.com/res/v1/web/search?q=brave+browser&count=5" \ -H "Accept: application/json" \ -H "Accept-Encoding: application/json" \ -H "X-Subscription-Token: brv_XXXXXXXXXXXXXXXXXXXX"

安全建議

  • 定期更新

    請定期檢查 Token 是否過期，並重新生成新的 API 金鑰。建議每 90-180 天更新一次。

  • 最小權限原則

    只勾選必要的權限，避免授予過多權限以減少安全風險。例如 GitLab 推薦 api、read_repository 與 write_repository。

  • 保密措施

    將 API 金鑰僅存於安全位置，不要在公開平台或程式碼中明文顯示。

  • 使用環境變數

    建議將所有金鑰存放於 .env 或 CI/CD 的 Secrets/Variables 中。

  • 加入 .gitignore

    確保包含金鑰的檔案被加入 .gitignore 以防止意外上傳到儲存庫。

  • 避免共享

    每個開發者應該使用自己的 API 金鑰，而不是共用一個 Token。這樣在人員離職時可以只撤銷特定金鑰。

常見問題 FAQ

我點錯頁面了，怎麼辦？
對於 GitLab，請確認您使用的是正確的連結 https://gitlab.com/-/user_settings/personal_access_tokens 而不是專案層級的 Deploy Tokens 頁面。對於 GitHub，請確保您在 Developer settings 下的 Personal access tokens 頁面。
Token 忘記複製了怎麼辦？
請重新產生新的 Token，舊的可以刪除或忽略。各平台都不會儲存您的 Token 明文，因此無法再次顯示。
是否需要開啟其他權限？
取決於您的使用需求。對於 GitLab，基本使用建議僅勾選 api、read_repository 與 write_repository。對於 GitHub，自動部署至少需要 repo 與 workflow。請遵循最小權限原則。
為什麼我的 Token 沒有作用？
可能是權限不足或 Token 已過期。請確認您選擇了正確的 Scopes，並檢查 Token 的有效期限。也可能是您的請求格式不正確，例如頭部格式或 API 端點錯誤。
API 金鑰適合存放在哪裡？
建議存放在：
  • 本地開發的 .env 檔案（確保已加入 .gitignore）
  • CI/CD 平台的 Secrets 或 Variables 設定中
  • 專業的密鑰管理服務，如 HashiCorp Vault 或 AWS Secrets Manager
GitHub 與 GitLab 的 Token 有何主要區別？
GitHub 提供更精細的權限控制（特別是 Fine-grained tokens），而 GitLab 的權限範圍較為廣泛。GitHub 的 Token 前綴為 ghp_，GitLab 的前綴為 glpat-。兩者在使用方式上類似，但 API 端點和請求格式有所不同。
