2. 資料庫、欄位與資料型態大解密

狼：一個資料庫就像一座大倉庫，裡面有許多「表格」充當儲物區， 欄位就像儲物格，能收納多種型態的資訊(๑•̀ω•́)ノ 以下是幾種常見分類：

  1. 數值資料 (Numeric Data)： 用於存放整數、浮點數、金額等，如 INT、FLOAT。
  2. 字串資料 (Character & Strings Data)： 儲存文字或符號，如 VARCHAR、TEXT。
  3. 日期/時間資料 (Date/Time Data) ：記錄日期或時間，如 DATE、DATETIME。
  4. 布林值 (Boolean Data) ：只有 True/False、Yes/No、1/0，簡單明瞭。
  5. 二進位字串 (Binary Data) ：可存圖片、檔案等二進位內容，但常只存路徑以免空間暴增。

兔：原來如此，每個儲物格都要放對型態，才能讓大倉庫運作更有效率！(ﾉ◕ヮ◕)ﾉ*:･ﾟ✧

2.3 資料庫常見約束條件

狼：接下來，就讓我這隻小狼再帶你更進一步認識幾個重要的關鍵字與約束條件， 讓你的資料表管理更順暢，也更能確保資料的正確性！

  1. PRIMARY KEY (主鍵)： 用來唯一識別該筆紀錄，如 id。不可重複，且用它就能快速定位一筆資料。
  2. AUTO_INCREMENT (自動遞增)： 讓數值欄位在每次新增資料時自動 +1，常跟 PRIMARY KEY 搭配。
  3. FOREIGN KEY (外鍵)： 用於連結其他表格的主鍵，幫助我們建立「表與表之間」的關聯。
  4. NOT NULL： 不允許此欄位為空值 (NULL)。
  5. UNIQUE： 保證該欄位內容不可重複 (例：Email 通常要唯一)。
  6. CHECK： 設定欄位必須符合某條件(部分資料庫支援度不同)， 例如年齡不能小於0。
  7. DEFAULT： 欄位預設值，若 INSERT 時沒給，就自動套用。

兔：哇～這樣就能好好管控資料，避免資料出現亂七八糟的情況了 (•̀ω•́)b

以下示範一組簡單的資料庫與表格建立流程：

-- 建立新的資料庫
CREATE DATABASE DemoDB;
USE DemoDB;

-- 建立「使用者」表 users，id 設為主鍵 + 自動增值
CREATE TABLE users (
    user_id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50),
    email VARCHAR(100)
);

-- 建立「訂單」表 orders，order_id 同樣是主鍵 + 自動增值
-- 同時用 user_id 作為外鍵，參照 users 表的 user_id
CREATE TABLE orders (
    order_id INT PRIMARY KEY AUTO_INCREMENT,
    order_date DATE,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);

-- 新增一些使用者
INSERT INTO users (name, email)
VALUES
('Alice', 'alice@example.com'),
('Bob',   'bob@example.com');

-- 新增一些訂單，並指定對應的 user_id
INSERT INTO orders (order_date, user_id)
VALUES
('2025-03-15', 1),
('2025-03-16', 2);

-- 查詢看看
SELECT * FROM users;
SELECT * FROM orders;

狼：看吧，小兔，這樣子我們就能同時維護好 主鍵(PRIMARY KEY)、 外鍵(FOREIGN KEY)， 還順便自動幫 id 編號 (AUTO_INCREMENT)， 管理資料就是這麼輕鬆♪ 

CREATE TABLE members (
    member_id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,  -- 不能空，且不得重複
    age INT CHECK(age >= 0),               -- 年齡不得小於 0
    join_date DATE DEFAULT CURRENT_DATE    -- 如果不給值，就用當天日期
);

狼：這些小管家能幫我們預先把好關，不讓怪怪的資料混進來。

2.4 狼兔劇場 - 資料表關聯

狼：最後來分享一下，表格和表格之間要是有關聯的話，該怎麼表達？(╯°ω°)╯
兔：對啊！像我們狼跟兔子是好朋友，可以互相認識，這樣的情況就是一種關聯對吧 (•̀ᄇ•́)و✧

一般常見有： 一對多 (One-to-Many)、 多對一 (Many-to-One)、 多對多 (Many-to-Many) 三種情境：

  1. 一對多 (One-to-Many)： 一隻狼可以對好幾隻兔子很感興趣 (同學會、同事關係之類)， 資料庫上就是「狼表」的主鍵 (wolf_id) 放在「兔表」裡當外鍵。
  2. 多對一 (Many-to-One)： 與「一對多」只是視角不同，有很多兔子歸屬同一隻狼～ 在「兔表」裡存 wolf_id 指向「狼表」的 id。
  3. 多對多 (Many-to-Many)： 若狼和兔子之間可以彼此互相認識、結成好搭檔， 就需要一張「中介表 (junction table)」來同時記錄狼與兔的 id。

例如下方簡單範例，建立 wolves 表與 rabbits 表，再用 wolf_id 來連結： 

-- 建立「狼表」
CREATE TABLE wolves (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50)
);

-- 建立「兔表」，使用 wolf_id 當外鍵
CREATE TABLE rabbits (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50),
  wolf_id INT,
  FOREIGN KEY (wolf_id) REFERENCES wolves(id)
);

狼：這樣就能知道每隻兔子跟哪隻狼有關係啦，資料在更新時也能保持連動，方便又安全！

3.9 刪除資料庫 / 清空內容

※ 警告：以下動作後，資料「不復返」(ಥ﹏ಥ)，請小心使用！

  1. 整個資料庫都砍掉 (Drop Database)
    DROP DATABASE Kitchen;
    這條下去後，整個 Kitchen（連同裡頭所有表格、資料）都會消失得無影無蹤。

  2. 只清除某張表的內容 (Truncate/Delete)
    TRUNCATE TABLE Ingredients; -- 咻地一聲全清光，ID歸零
    DELETE FROM Ingredients;      -- 一筆筆刪除，但自動增值計數器保留
    用於測試資料或清空表格時很好用，不過千萬別在正式環境中「手滑」(>ω<)！

<?php

// 1) 新增 INSERT

// 2) 查詢 SELECT
$sql_select = "SELECT * FROM Ingredients";

while ($row = mysqli_fetch_assoc($result)) {
    echo "ID: ".$row['id']." / 名稱: ".$row['name']." / 數量: ".$row['quantity']." / 到期日: ".$row['expiry']."
";

mysqli_query($conn, $sql_update);

// 4) 刪除 DELETE
$sql_delete = "DELETE FROM Ingredients 
               WHERE name = 'Sugar'";
mysqli_query($conn, $sql_delete);

// 關閉連線
mysqli_close($conn);
?>

7. 進階 SQL 功能與語法探索

          

            狼：進階世界的資料操作可就更多花招囉，一起來瞧瞧
            JOIN、GROUP BY/HAVING、UNION、
            INDEX、Transaction 等功能吧 (๑•̀ω•́)و✧
          


          
7.1 關聯查詢 (JOIN)

          

            INNER JOIN：只取得符合條件的兩表交集部分。

            LEFT JOIN：以左表為主，無論是否符合條件，左表保留。

            RIGHT JOIN：以右表為主，右表保留。

            FULL JOIN (有些資料庫支援)：左右表都列出。
          

          

            
            
-- 假設已存在 users 表 (主鍵 user_id)， orders 表 (外鍵 user_id)
SELECT 
  users.name AS user_name,
  orders.order_date
FROM users
INNER JOIN orders ON users.user_id = orders.user_id;

          

          

            狼：INNER JOIN
            就像把「兩邊對得上號」的資料配對起來，沒有對應關係的就忽略。

            兔：好比同學會名單與繳費清單，沒繳費就不會出現在最終名單裡 (๑•́ ₃ •̀๑)
          


          
7.2 群組與篩選 (GROUP BY / HAVING)

          

            如果想要針對「同一個屬性」做分組，並且計算平均、總和或統計，就需要
            GROUP BY；若要再進一步篩選「分組後」的結果，使用
            HAVING (作用在群組層級)。
          

          

            
            
-- 統計每個使用者的訂單數量
SELECT 
  users.name AS user_name, 
  COUNT(orders.order_id) AS total_orders
FROM users
LEFT JOIN orders ON users.user_id = orders.user_id
GROUP BY users.user_id, users.name
HAVING COUNT(orders.order_id) > 0;  -- 只列出有下訂單的人


          

          

            狼：GROUP BY 對結果做「團體照」，接著用各種函式 (COUNT,
            SUM, AVG...) 拿到群組資訊。

            兔：HAVING 就是針對合照後，再決定哪些群組要被留下來。
          


          
7.3 DISTINCT / UNION：去重 & 合併

          

            DISTINCT：去除結果中的重複列，保留唯一值。

            UNION / UNION ALL： 將兩個查詢結果合併；UNION
            會自動去重，UNION ALL 不去重。
          

          

            
            
-- 假設一張表是 "舊會員名單"、另一張是 "新會員名單"
-- 想合併查詢，且自動移除重複者，就用 UNION
SELECT name, email FROM old_members
UNION
SELECT name, email FROM new_members;

-- 如果想保留所有重複，則用 UNION ALL


          

          

            狼：DISTINCT 和
            UNION 就是「不要重複」的小神器，平常只用 SELECT
            時尤其好用。

            兔：像參加活動的名單，有重複報名者，用這招就一眼看出來啦 ( • ̀ω•́ )✧
          


          
7.4 索引 (INDEX)

          

            想要加速查詢，就在常被搜尋或比對的欄位上加 INDEX。
            這就像圖書館裡的「目錄索引」，找書就能飛快定位到正確區域，不用一本一本翻找。
          

          

            
            
-- 建立索引
CREATE INDEX idx_users_email ON users (email);

-- 或者在建立表時直接加
CREATE TABLE users (
    user_id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50),
    email VARCHAR(100),
    INDEX (email)
);

          

          

            狼：給經常用來搜尋或排序的欄位加索引，就能大幅提升查詢效率，但要注意寫入、更新也會稍微變慢。

            兔：就像給圖書館書本上好索引卡，找到書好快，可是新增新書也要花時間建卡片
            (ﾉ◕ヮ◕)ﾉ*:･ﾟ✧
          


          
7.5 交易控制 (Transaction)

          

            有些操作要嘛全數成功、要嘛全部取消，避免「半途失敗」產生爛帳。這時就需要用
            Transaction 搭配 COMMIT 與 ROLLBACK。
          

          

            
            
START TRANSACTION;          -- 或 BEGIN;
  UPDATE accounts SET balance = balance - 500 WHERE user_id = 1;
  UPDATE accounts SET balance = balance + 500 WHERE user_id = 2;
COMMIT;   -- 確認都成功後送出

-- 如果中間出錯，可用
ROLLBACK; -- 會回到 START TRANSACTION 前的狀態


          

          

            狼：像轉帳動作，要保證「甲戶扣款」和「乙戶入帳」同時成功，不然就全部不算！

            兔：不然半路壞掉，錢憑空消失，就尷尬啦 ╮(╯_╰)╭
          


          
7.6 總結

          

            JOIN 讓我們把多張表的資料「湊成一塊」。

            GROUP BY / HAVING 幫忙做統計及群組過濾。

            DISTINCT / UNION 處理重複資料或合併查詢。

            INDEX 提高查詢效率。

            Transaction + COMMIT / ROLLBACK
            確保關鍵操作的完整性。

            Constraints (NOT NULL, UNIQUE, CHECK, DEFAULT)
            加強資料正確性。
          

          

            狼：這些進階 SQL 功能和語法可以讓我們處理更複雜的資料庫需求，
            若還想更深入，可以再研究「視圖 (VIEW)」、「觸發器 (TRIGGER)」、
            「儲存程序 (Stored Procedure)」、「函式 (Function)」等唷 (⁎˃ᴗ˂⁎)♪

            兔：哈哈！感謝小狼帶路～ 這些知識真是太好用，來繼續冒險吧 (✿ﾟ▽ﾟ)ノ
