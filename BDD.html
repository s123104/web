Given-When-Then語法規則

Gherkin語法是BDD的核心工具，使用自然語言結構化描述系統行為， 讓技術和非技術人員都能理解和參與。

基本結構

Feature文檔結構 
Feature: [功能名稱]
  As a [角色]
  I want [目標]
  So that [價值]

Scenario: [場景名稱]
  Given [前置條件]
  When [執行動作]
  Then [預期結果]
G

Given

前置條件

  • • 描述系統的初始狀態
  • • 設定測試環境和資料
  • • 可以有多個Given條件
W

When

執行動作

  • • 描述用戶執行的操作
  • • 通常只有一個When動作
  • • 觸發系統行為的事件
T

Then

預期結果

  • • 描述期望的系統回應
  • • 可以有多個Then結果
  • • 可驗證的斷言

實際範例

用戶登入系統功能 
Feature: 用戶登入系統
  作為網站用戶
  我希望能夠登入系統
  以便訪問個人化內容

  Background:
    Given 系統已啟動
    And 用戶資料庫包含測試用戶

  Scenario: 成功登入
    Given 我在登入頁面
    When 我輸入正確的用戶名和密碼
    And 我點擊登入按鈕
    Then 我應該被導向到首頁
    And 我應該看到歡迎訊息

  Scenario: 登入失敗
    Given 我在登入頁面
    When 我輸入錯誤的密碼
    And 我點擊登入按鈕
    Then 我應該看到錯誤訊息
    And 我應該停留在登入頁面

撰寫最佳實踐

遵循專業的撰寫規範和設計原則，確保BDD場景清晰易懂、 可維護且能有效促進團隊協作。

語言使用原則

使用業務領域語言

避免技術術語，採用業務人員熟悉的詞彙

保持簡潔明確

使用清晰直接的表達方式，避免冗長描述

使用主動語態

明確指出動作的執行者和操作內容

避免實作細節

專注於行為結果而非技術實現方式

場景設計原則

一個場景一個規則

每個場景專注測試一個業務規則

場景獨立性

場景應該獨立且可重複執行

有意義的命名

使用具描述性的場景名稱

專注行為

關注系統行為而非具體實現

參數化場景範例

Scenario Outline 參數化測試 
Scenario Outline: 多種登入嘗試
  Given 我在登入頁面
  When 我輸入用戶名 "<username>" 和密碼 "<password>"
  And 我點擊登入按鈕
  Then 我應該看到 "<result>"

  Examples:
    | username    | password    | result       |
    | valid_user  | valid_pass  | 歡迎訊息     |
    | valid_user  | wrong_pass  | 密碼錯誤     |
    | wrong_user  | valid_pass  | 用戶不存在   |
    | empty       | empty       | 請填寫必填欄位 |
